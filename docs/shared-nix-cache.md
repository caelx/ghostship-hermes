# Shared Nix Cache

`ghostship-hermes` phase 1 shared-cache support uses `caelx/ghostship-cache` as a signed GHCR-backed Nix binary cache. The image build contract does not change: `publish-image` still builds the explicit `ghostship-hermes-image` bundle on the runner host and exports it with `scripts/export_publishable_image.sh`.

## Workflow Contract

- before `nix build`, `publish-image` consumes the shared cache only when `caelx/ghostship-cache` already has a reachable `cache-index`
- on a cold cache with no index yet, normal push builds continue with the uncached host-side build and do not try to refresh the cache inline
- shared-cache publication is gated to the daily scheduled `publish-image` run at `14:00 UTC` or an explicit `workflow_dispatch` run with `publish_shared_cache=true`
- if the proxy is unavailable after an index is expected, the workflow continues with the normal uncached host-side build
- if the proxy serves a signature mismatch, the Nix build fails; the workflow does not disable signature verification
- when a cache-refresh run is requested, the workflow captures the upload plan before the real build and then signs and uploads only those planned store paths after the image build succeeds
- a run only counts as a real cache seed when it creates or updates the OCI `cache-index`; a successful upload step with zero planned paths is a no-op, not a seeded cache

## Required GitHub Actions Configuration

Actions variables:

- `GHOSTSHIP_CACHE_PUBLIC_KEY`: shared cache public key string
- `GHOSTSHIP_CACHE_REPO`: optional override; defaults to `caelx/ghostship-cache`

Actions secrets:

- `GHOSTSHIP_CACHE_GHCR_TOKEN`: optional override token for cache publication. In GitHub Actions, the workflow now prefers the job `GITHUB_TOKEN` path first and only falls back to this override when needed. Any override still needs package-write access to `ghcr.io/caelx/ghostship-cache/nix-cache`.
- `GHOSTSHIP_CACHE_SIGNING_KEY`: private key contents generated by `nix-store --generate-binary-cache-key`

## Verification

### Cold-cache publish still succeeds

1. Point `GHOSTSHIP_CACHE_REPO` at an empty backend or clear the existing `nix-cache` package index.
2. Trigger the daily scheduled run or dispatch `publish-image` on `main` with `publish_shared_cache=true`.
3. Expect the image build to complete successfully even though the cache has no reusable Ghostship paths yet.
4. Expect the pre-build cache planning step to produce a non-empty upload plan and the later publish step to upload those paths into `ghcr.io/caelx/ghostship-cache/nix-cache`.
5. Confirm a `cache-index` manifest now exists for `ghcr.io/caelx/ghostship-cache/nix-cache`; without that manifest, later runs will still skip cache consumption.

### Warm repeat publish reuses cached paths

1. Trigger `publish-image` again without changing image inputs and without `publish_shared_cache=true`.
2. Confirm `Bootstrap shared Ghostship Nix cache` no longer reports `cache index not present yet`; it should start the local proxy instead.
3. Inspect the image build log for cache-backed reuse, such as proxy-backed substitutions from `http://127.0.0.1:37515` or materially reduced local build work.
4. Inspect the shared-cache publish summary; it should report few or zero newly uploaded paths.
